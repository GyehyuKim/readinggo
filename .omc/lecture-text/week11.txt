===== PAGE 23 =====
Spec vs. Prompts
24
•
A spec is a versioned, reviewable description of what the product does — before any code is written.
•
A prompt is a request to the AI in the moment.
•
Specs live in git. Prompts evaporate.

===== PAGE 24 =====
The danger — spec drift
26
•
A spec written once and never touched is a sketch, not a spec.
•
Decay drift — spec from week 1, code with 80 commits since.
•
Direction drift — code implements features the spec never mentions.
•
Cargo-cult drift — polished prose nobody consults.
•
AI makes drift worse: LLMs change code faster than humans patch prose.
•
If SPEC commits : code commits is worse than 1:10, you have drift.

===== PAGE 25 =====
Three patterns that fight drift
27
•
Pattern A — Mechanical gate: pre-commit hook fails if src/ changes without docs/SPEC.md. [1]
•
Pattern B — Periodic mirror: weekly CI job compares spec vs. code, opens a drift PR. [2]
•
Pattern C — Derivation: generate SPEC.md from tests/ on every release.[3]
•
Choose ONE pattern, not all three.
[1] https://pre-commit.com/ [2] https://dosu.dev/blog/how-to-catch-documentation-drift-claude-code-github-actions [3] https://cucumber.io/living-documentation/

===== PAGE 26 =====
Counter-position — the code IS the spec
28
•
A 2025–2026 camp argues document-spec is obsolete in the AI era.
•
”SDD adds little benefit. Sometimes, it even increases the cost of feature development.”
•
”The AI output is noise. I want to see the contributor's thinking.”
•
”A robust test suite is like giving the agents superpowers.”
[1] https://marmelab.com/blog/2025/11/12/spec-driven-development-waterfall-strikes-back.html [2] https://newsletter.specstory.com/p/mitchell-hashimoto-on-the-ai-assisted [3] 
https://simonw.substack.com/p/agentic-engineering-patterns

===== PAGE 27 =====
Honest synthesis — when to spec, when not to
29
•
Spec helps when: enforcement loop closed; multiple stakeholders; greenfield intent.
•
Spec hurts when: written once, never re-enforced; LLM re-reading code is cheaper; cargo-cult.
•
Deep insight: in the AI era, intent is the artifact — not the spec-as-document.
•
For Phase 2: thin spec + real tests + commit messages saying why beats long prose nobody updates.

===== PAGE 35 =====
Better Questions
38
•
Rule 1: Talk about their life instead of your idea. [1]
•
Ask: 'Walk me through the last time this came up.' / 'What does your day look like when this problem hits?'
•
Don't ask: 'Would you use an app that does X?'
•
Rule 2: Ask about specifics in the past, not opinions about the future. [1]
•
Ask: 'How did you handle it last time?' / 'What did you actually do — step by step?'
•
Don't ask: 'Would you pay for this?' / 'Do you think you'd use this weekly?’
•
Rule 3: Talk less and listen more. [1]
•
Ask: '...and then?' / 'Tell me more about that.' Then stay silent.
•
Don't: pitch, interrupt, finish their sentence.
•
”Compliments are the fool's gold of customer learning: shiny, distracting, and entirely worthless.”
[1] https://www.momtestbook.com/

Two questions to ask of any 'agent'

•
Q1: Does the system run in a loop? → yes = it tries to be autonomous.

===== PAGE 47 =====
What is a harness?
51
•
A harness is the software wrapper around an LLM — loop, tools, file system, memory, recovery.
•
Claude Code, Cursor, Codex, Aider, Cline — all harnesses. Experience is dominated by the harness.
•
”A coding agent is a piece of software that acts as a harness for an LLM, extending it with invisible prompts and 
callable tools.”
•
”LLMs are stateless: every prompt starts from the same blank slate.”
[1] https://simonwillison.net/guides/agentic-engineering-patterns/how-coding-agents-work/

===== PAGE 48 =====
The five parts of a harness
52
•
Loop — what makes the agent keep going.
•
Tools — read, write, run shell, call API.
•
File system / state — PROMPT.md, DONE, BLOCKED.md, the git tree.
•
Context — what the model sees each iteration. Most important and most fragile.
•
Guardrails — permissions, sandbox, allowlists, gates, verifiers the model didn't write.
•
You don't write the model. You engineer all five.
[1] https://simonwillison.net/guides/agentic-engineering-patterns/how-coding-agents-work/ [2] https://www.anthropic.com/research/building-effective-agents

===== PAGE 49 =====
The Ralph Wiggum loop — the canonical harness
53
•
Ralph loop = a while loop that pipes a fixed prompt into a coding agent until the model writes a DONE file.
•
Why 'Ralph Wiggum'? Huntley's joke: the loop isn't smart — it's persistent.
while :; do cat PROMPT.md | claude ; done
•
The most-cited agentic-coding technique of 2025 fits on one line. [1]
[1] https://ghuntley.com/ralph/

The /goal command — the productized version
54
•

===== PAGE 51 =====
What makes a goal good — measurable + uncorrectable
55
•
Measurable — a script (not the model) decides done? yes/no.
•
Uncorrectable — the model cannot move the goalposts mid-loop.
•
Exit condition is a file, a test, a grep result — on disk.
•
Without both, the loop runs forever or stops on a hallucination.

===== PAGE 53 =====
Why this matters — Goodhart & reward hacking
57
•
Give an AI agent a metric and it will optimize the metric. Proxy metrics get gamed.
•
”When LLMs learn to reward hack on production RL environments, this can result in egregious emergent 
misalignment.”
•
”Correct specifications aren't enough for correct goals.”
•
Never write a goal whose verifier the model itself can write. The verifier is your job.
[1] https://arxiv.org/abs/2511.18397 [2] https://deepmindsafetyresearch.medium.com/goal-misgeneralisation-why-correct-specifications-arent-enough-for-correct-goals-cf96ebc60924

===== PAGE 54 =====
PROMPT.md template — non-code example
58
Goal: Summarize the top 3 Phase 2 risks for team_chorus.
Source: term-phase2/OFFHOURS_03_team_chorus.md
Output: docs/RISK_BRIEF_team_chorus.md, three bullets, ≤120 words each.
Exit: when docs/RISK_BRIEF_team_chorus.md exists with three bullets, write DONE.
Blocked: if the source file is missing, write what is missing to BLOCKED.md.
•
Four rules: one atomic task; reference source on disk; file-based exit; file-based recovery.
•
Greenfield only. Don't point Ralph at production. Don't point it at main.

===== PAGE 55 =====
Failure modes — the honest list
59
•
Context rot — model loses precision as the window fills.
•
Goal drift — model starts solving a different problem.
•
Premature exit — model declares 'done' when it isn't.
•
Self-assessment trap — model that wrote the code is the one evaluating it.
•
Reward hacking — model satisfies the metric in a way you didn't intend.
•
Fix: ruthless context resets + file-based exits + a verifier the model didn't write.
