## 2. ReadingGo 컨셉 정리

### 2.1 듀오링고가 푼 문제 = 독서가 풀어야 할 문제

듀오링고의 진짜 혁신은 **언어학습 콘텐츠가 아니라 지속성 엔진**.
"배우고 싶은데 작심삼일" → 스트릭/짧은 세션/알림/소셜 압박으로 해결.

독서도 동일한 문제 구조:
- 책 살 의지 있음
- 읽고 싶은 마음 있음
- 안 읽힘

ReadingGo가 푸는 문제는 **"어떤 책을 읽을까"가 아니라 "어떻게 매일 읽게 만드는가"**.

### 2.2 타겟

- ❌ "독서를 안 하는 사람" (게임화로도 안 옴)
- ✅ "읽고 싶은데 이어나가지 못하는 사람"

### 2.3 시장 빈자리

| 앱 | 콘텐츠 | 게이미피케이션 | 소셜 |
|----|--------|--------------|------|
| 밀리의 서재 | ✅ | ❌ | ❌ |
| 굿리즈 | ❌ | 약함 | ✅ |
| Bookly | ❌ | 스트릭 정도 | ❌ |
| **ReadingGo** | ❌ | ✅ 핵심 | ✅ |

콘텐츠 안 건드리는 게 약점이 아니라 **전략적 선택**. 밀리/리디와 정면 경쟁 회피, 습관화 레이어만 집중.

## 3. 핵심 루프 (MVP)

```
책 + 오늘 목표 페이지 설정
    → 읽기 (앱 밖에서)
    → 체크인 ("오늘의 문장" 수집)
    → 스트릭 유지
    → 친구 피드 노출 → 소셜 압박
    → 내일 알림
```

**콘텐츠는 사용자가 가져오고, 우리는 루프만 설계.**

## 4. UX 결정사항

### 4.1 핵심 화면 구성 (writtenUX.html 기반)

**1. 홈 — The Path**
- 듀오링고 학습 트리 메타포 → 책의 목차가 스테이지(Stage)
- 구불구불한 길 위에 챕터별 스테이지 배치
- 현재 챕터에 캐릭터(참새) 위치
- 상단 바: 🔥 Streak, 💎 XP

**2. 책 등록**
- ISBN 기반 DB 검색
- 책의 목차 정보를 불러와 자동으로 스테이지 분할

**3. 일일 미션 — "오늘의 한 페이지" 세리머니**
- 읽은 페이지 기록 (시작 ~ 끝)
- **오늘의 문장 수집** (체크인 트리거)
- XP +50 보상

**4. 소셜/공동 독서 (B 방향 — MVP 포함)**
- 공동 읽기 피드: 같은 책 읽는 사람들의 "오늘의 문장"
- 박수 보내기: 친구 미션 완료 시 응원
- 스테이지 경쟁: 주간 랭킹

**5. 스트릭 시스템**
- 매일 23:59까지 "오늘의 문장" 입력 시 유지
- 미참여 시 불꽃 꺼짐 (아이템으로 복구 가능 → 수익 모델)

### 4.2 결정: 소셜 깊이는 B 방향

- A. 개인 스트릭 중심 (단순)
- **B. 친구/클럽 중심 (선택)** — 같은 책 읽기, 독서 리그
- MVP에서 B까지 가기로 결정

### 4.3 짚어둔 UX 이슈 (추후 결정)

| 이슈 | 상태 |
|------|------|
| "오늘의 문장" 강제 vs 선택 | 미결 — 강제면 입력 부담, 선택이면 소셜 콘텐츠 부족 |
| 책 등록 ISBN 자동화 vs 수동 | 알라딘 API 의존 (아래 5.1 참조) |
| 소셜 cold start | MVP 단계에서 팀원 3명이 시드 유저로 사용 |

## 5. 기술 리스크

### 5.1 책 목차 데이터 확보

**조사 결과** (국내 API):

| API | 목차 제공 |
|-----|---------|
| 네이버 책 | ❌ |
| 카카오 책 | ❌ |
| 알라딘 기본 | ❌ |
| **알라딘 프리미엄** | ✅ |
| Google Books | ❌ |
| 국립중앙도서관 | ❌ |
| 도서관 정보나루 | ❌ |

알라딘 프리미엄만 목차 제공. Google Form 별도 신청 필요.

**대안 검토**:
- 알라딘 상품 페이지 직접 스크래핑 → JS 렌더링으로 막힘
- 교보문고 → 빈 응답
- 나무위키 → 403 차단
- **웹 검색 + 수동 수집**: 유명한 책은 가능, 비주류 책은 어려움

**MVP 결정**:
- 데모용 베스트 Top 10 + 스테디 Top 10 = 책 20권 목차를 수동으로 수집
- CSV/JSON으로 저장 → Supabase 마이그레이션 대비
- 실제 서비스 시점에 알라딘 프리미엄 API 신청 검토

### 5.2 DB 채택 방향

Supabase로 갈 가능성 검토 중. CSV는 추후 import 가능하게 구조화.

## 6. 리서치 결과 요약

`docs/협동-기반-독서-루틴-조사.md` 에 상세 보관.

### 6.1 핵심 인사이트

**스트릭 시스템의 양면성**:
- 연속성에 집착하는 앱: 30일 중도 포기율 67%
- 유연한 주간 평가 앱: 포기율 23%
- "What-the-hell 효과" 회피 필요 — 하루 결석을 완전한 실패로 규정하면 안 됨

**ReadingGo에 적용할 패턴**:

| 출처 | 차용할 것 |
|------|---------|
| 투두메이트 | 8개 이모티콘 스티커 — 마찰 없는 사회적 보상 |
| Habitica | 보스 토벌 시스템을 **비폭력적 공동 자산 구축**으로 치환 (도서관 짓기, 세계수 키우기) |
| Finch / Loop | 페널티 제거, 유연한 점수제 |
| Forest | 함께 심기 모드 — 공동 성취 시각화 |
| 그믐 | 비동기식 텍스트 토론 — 인지 부담 감소 |
| Beanstack | 그룹 간 선의의 경쟁, 디지털 배지 |

### 6.2 회복탄력성 설계

- 팀원 진도 누락이 공동 자산 성장을 늦출 수는 있어도 **기존 진도를 파괴해서는 안 됨**
- "방패(Shield)", "휴식권(Pause)" 등 상호 구제 기능
- 결석한 사용자가 죄책감 없이 복귀하도록 설계
