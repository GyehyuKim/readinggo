# UX-AUDIT.md — ReadingGo UX 감사 체크리스트

> UX 개선을 감(感)이 아니라 **반복 가능한 감사 절차**로 돌리기 위한 렌즈 모음(#1219).
> 표준 방법론(닐슨 휴리스틱 · 인지적 워크스루 · 저니 QA · 모바일 에르고노믹스)을 ReadingGo 언어로 번역했다.
> **사용법**: §3 저니를 실제로 걸으면서 §2 휴리스틱 위반을 기록 → §5 심각도로 정렬 → 이슈화.
> 시각·토큰 규칙의 정본은 [`DESIGN.md`](./DESIGN.md) — 이 문서는 그 위의 **행동·흐름** 레이어.

## 0. 캘리브레이션 정답지 — "좋은 감사라면 이걸 잡았어야 한다"

과거 실사용자가 실제로 걸려 넘어진 것들. **감사 한 바퀴를 돌고 나서 이 클래스의 문제를 하나도 못 찾았다면, 감사가 아니라 산책을 한 것이다.** 새 감사 결과는 늘 이 목록과 대조해 렌즈를 벼린다.

| 실사례 | 위반 렌즈 | 교훈 |
|---|---|---|
| 잘못 담은 책을 지울 수 없음 — '중단'만 존재 (#1195) | H3 사용자 통제 | 되돌리기 없는 액션은 전부 의심 |
| 새 책을 등록해도 이전 책이 홈에 그대로 — 전환 화살표를 "겨우 찾음" (#1196) | H6 인식>회상, 워크스루 Q2 | 발견성은 "있다"가 아니라 "찾는다" |
| 입력 탭 시 스크롤 점프 · 중첩 스크롤 (#1197) | §4 에르고노믹스 | <16px 입력 = iOS 자동 줌 |
| 문장 10개 남기면 세리머니 10번 — 간섭 과다 (#1198) | H7 효율, H8 미니멀 | 반복 사용자의 동선을 별도로 걷기 |
| Safari 뒤로가기가 로그인으로 튕김 (#1199) | H3 통제, 플랫폼 관례 | 브라우저/OS 제스처도 UI다 |
| 재키 말투가 "킹받음" (#1200) | H2 현실 일치, Feeling Guardrail | 카피·톤도 감사 대상 |
| OCR 추출 중 무반응 — 멈춘 줄 앎 (#1201) | H1 상태 가시성 | 2.2s 토스트는 장시간 작업 피드백이 아님 |
| 완독한 책이 홈에 '읽는 중'으로 계속 표시 (#1217) | H4 일관성(상태 불변식) | 같은 전이(중단·삭제·완독)는 같은 규칙 |

## 1. 감사 운영법

- **도구**: 라이브 데모(`https://readinggo.hyuniverse.workers.dev`)를 실제 브라우저로 조작(claude-in-chrome). 스크린샷 = 증거.
- **계정**: 프로덕션은 실사용자 데이터 — **게스트 플로우 중심 + 읽기 위주**. 쓰기 검증은 일회용 테스트 계정(끝나면 삭제).
- **뷰포트**: 모바일 390×844 기본(실사용 다수) + 데스크탑 1회.
- **주기**: 릴리스 전 = §3 P2 저니 + 홈 휴리스틱(축소판). 월 1회 = 전면(§2×전 화면 + §3 전 페르소나).
- **기록**: 발견 즉시 `화면 · 재현 단계 · 위반 렌즈 · 심각도 · 스크린샷 · 제안` 형식으로. 감사 끝에 §5 규칙으로 이슈화.
- **자동 캡처 스윕 (#1243)**: `cd docs/readinggo && npm run audit:ui` — 375px 실뷰포트로 신규·재방문 게스트 두 패스(홈·검색 3상태·세리머니(+지연 오버레이)·3탭·책 상세)를 스크린샷 + 콘솔·네트워크 에러로 일괄 수집한다(`AUDIT_URL`/`AUDIT_OUT` 환경변수로 대상·산출 변경). **도구는 판정하지 않는다** — 캡처를 §2 렌즈로 사람/에이전트가 보는 것까지가 감사다. 브라우저 수동 감사(claude-in-chrome)는 인터랙션 상태(hover·포커스·파괴적 액션 확인창) 보완용으로 병행.
- **스크린샷 라운드트립(픽스 검증)**: 프론트 변경 PR은 변경 화면의 수정 후 캡처를 실측해 PR 본문에 증거로 남긴다 — "코드가 맞아 보임"이 아니라 "렌더가 맞음"으로 닫는다(2026-07-13 감사 픽스 #1240~#1242 선례).

## 2. 휴리스틱 체크리스트 (닐슨 10 → ReadingGo 번역)

각 항목은 **질문 형태**다. 화면마다 "아니오"가 나오면 기록한다.

- **H1 상태 가시성** — 기다리게 하는 모든 순간(OCR·LLM·저장·로딩)에 진행 표시가 있는가? 완료·실패가 명시적으로 보이는가? *(무반응 = 자동 심각도 3)*
- **H2 현실 세계 일치** — 화면의 말이 독자의 말인가? (둥지·짹·한 문장 은유의 일관성 / `user_book`·`upsert` 류 시스템 언어 노출 금지 / 재키의 톤이 "같이 읽은 친구"인가)
- **H3 사용자 통제·자유** — 모든 액션에 출구가 있는가? (취소·되돌리기·삭제 / 뒤로가기=모달 닫기 / 실수 복구 경로) *"어쩌다 눌렀는데"로 시작하는 시나리오를 최소 3개 시도해 본다.*
- **H4 일관성·표준** — 같은 의미는 같은 모양인가? (버튼 위계 = DESIGN.md 1차 솔리드/2차 tonal/3차 텍스트 / 상태 전이 불변식: 중단·삭제·완독·재개가 active 책을 같은 규칙으로 다루는가 / 아이콘↔라벨 의미 일치)
- **H5 오류 예방** — 실수를 하기 어려운 구조인가? (파괴적 액션 확인창 / 빈 입력 제출 방지 / 잘못된 대상에 기록될 가능성)
- **H6 인식 > 회상** — 기능이 "보이는" 곳에 있는가, 외워야 하는가? (핵심 컨트롤이 아이콘만으로 숨어 있지 않은가 / 신규 사용자가 라벨 없이 의미를 추측해야 하는 컨트롤 수 세기)
- **H7 유연성·효율** — 매일 쓰는 동선의 탭 수를 세라. (체크인까지 몇 탭? 문장 3개 남기기까지? / 반복 사용자용 지름길이 신규 사용자를 방해하지 않는가)
- **H8 미학·미니멀** — 화면에 지금 과업과 무관한 요소가 몇 개인가? + **Feeling Guardrail**(DESIGN.md): Duolingo식 압박·SaaS 대시보드·과한 폭죽·숫자/랭킹 중심이 스며들지 않았는가?
- **H9 오류 인지·복구** — 에러가 났을 때 (a) 났다는 사실 (b) 왜 (c) 어떻게 복구를 다 말해주는가? ("잠시 후 다시"는 (a)만 충족 — 반복 실패 시 다른 안내가 있는가)
- **H10 도움말·빈 상태** — 모든 빈 상태(empty state)가 **다음 행동을 가르치는가**? (빈 서재·빈 피드·문장 0개 — placeholder가 아니라 CTA인가)

## 3. 저니 QA — 페르소나 3 + 인지적 워크스루

각 저니를 걸을 때 **매 단계 4질문**(인지적 워크스루)을 던진다:
① 지금 뭘 해야 하는지 화면이 말해주는가? ② 그걸 할 컨트롤을 찾을 수 있는가? ③ 누르면 무슨 일이 일어날지 예상되는가? ④ 결과가 "잘 됐다"고 확인해주는가?

### P1 — 첫 방문 게스트 (설치·로그인 없음)
랜딩 → 첫 화면 이해(3초: 뭐 하는 앱인지) → 둘러보기 → 책 검색 → 게스트로 문장 남기기 → **가입 전환점**(왜 로그인해야 하는지 설득되는가) → 로그인 후 게스트 기록이 살아남는가.

### P2 — 갓 로그인한 신규 (핵심 루프 최초 1회)
온보딩 → 첫 책 등록(검색·바코드·직접) → 첫 문장(직접 입력·사진 OCR) → 첫 체크인·세리머니 → 홈 복귀 상태 확인 → **다음 날 다시 올 이유가 화면에 있는가**.

### P3 — 3주차 습관 사용자 (반복 동선의 마찰)
재방문 → 체크인까지 탭 수 → 문장 여러 개(배치) → 책 전환·완독 처리 → 피드·마을 눈팅 → 프로필/서재 관리(삭제·중단·재개). *이 저니에서는 H7(효율)에 가중치.*

## 4. 모바일 에르고노믹스 체크리스트

- [ ] 터치 타깃 **≥44×44px** (아이콘 버튼 전수 — 실측)
- [ ] 핵심 CTA가 **엄지 존**(화면 하단 1/3)에 있는가; 상단 모서리에 자주 쓰는 액션이 없는가
- [ ] 텍스트 입력 **실효 16px 이상** (iOS 자동 줌 방지 — #1197 전역 플로어 회귀 감시)
- [ ] 키보드가 올라왔을 때 입력창·제출 버튼이 가리지 않는가
- [ ] 시트/모달 내부 스크롤이 배경으로 새지 않는가 (`overscroll-behavior`)
- [ ] safe-area (노치·홈 인디케이터) 침범 없음
- [ ] 뒤로가기 제스처 = 최상위 오버레이 닫기 (#1199 회귀 감시)
- [ ] 가로 스크롤이 페이지 전체에 생기지 않는가

## 5. 판정·이슈화 규칙

**심각도** (닐슨 스케일):

| 등급 | 의미 | 처리 |
|---|---|---|
| 4 | 과업 차단 — 진행 불가·데이터 유실 | 즉시 이슈 + P0/P1 |
| 3 | 과업 실패 유발 — 우회 어려움·이탈 예상 | 즉시 이슈 |
| 2 | 마찰 — 우회 가능하나 짜증 | 묶어서 이슈(화면/테마 단위) |
| 1 | 미관·사소 | 감사 보고서에만, 폴리시 배치 대기 |

**우선순위 = 심각도 × 노출 빈도.** 핵심 루프(P2·P3 저니) 위의 심각도 2는 주변 화면의 3보다 먼저다.

**보고 형식** (이슈 본문):

```text
화면/저니: P2 첫 체크인
재현: ① … ② …
위반: H1 상태 가시성 · 심각도 3
증거: (스크린샷)
제안: …
```
